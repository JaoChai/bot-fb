# VIP ข้ามการแจ้งเตือน Support ล่าช้า — Design

**วันที่:** 2026-07-20
**เป้าหมาย:** ลูกค้า VIP ไม่ต้องเจอขั้นแจ้งเตือน "ทีม Support อาจใช้เวลานานกว่าปกติ" (`<support_delay_warning>`) — ยืนยันออเดอร์ปุ๊บไปขั้นจ่ายเงินทันที ลูกค้าทั่วไปคงเดิมทุกอย่าง

## บริบท

- System prompt ของ bot 26 อยู่ใน Neon DB: ตาราง `flows` id=24 (v25, ~41.5k chars) — project `bot-facebook` (`solitary-math-34010034`)
- Section `<support_delay_warning status="ACTIVE">` บังคับแจ้งเตือนหลัง Step 2 (Confirm) เมื่อตะกร้ามี Nolimit Level Up+ และต้องรอลูกค้าตอบ "ตกลง" ก่อนไปต่อ โดยมีกฎ `⛔ ทั้ง VIP และลูกค้าทั่วไป ต้องแจ้งเหมือนกัน`
- ระบบรู้จัก VIP ผ่าน Memory Notes (`ลูกค้า VIP — ซื้อยืนยันแล้ว X ครั้ง...` จาก VipDetectionService) และ prompt มี `<priority_1_vip>` VIP Protocol ที่ให้ VIP ข้าม Terms ได้อยู่แล้ว — ใช้กลไกเดียวกัน

## การตัดสินใจ (ยืนยันจากเจ้าของแล้ว)

- VIP **ข้ามขั้นนี้ทั้งหมด** ไม่แจ้งอะไรเลย (ไม่ใช่แจ้งแบบสั้น)
- แก้เฉพาะ prompt ใน DB — ไม่แตะโค้ด backend (แนวทาง A)
- ถ้า LLM ไม่เห็น memory VIP → ปฏิบัติเหมือนลูกค้าทั่วไป (fallback ปลอดภัย = พฤติกรรมปัจจุบัน)

## การแก้ไข — 3 จุดใน `<support_delay_warning>` ของ flow 24

### จุดที่ 1: TRIGGER — เพิ่มข้อยกเว้น VIP

เดิม:
```
TRIGGER: หลัง Step 2 (Confirm) ลูกค้ายืนยันรายการแล้ว
→ ถ้าในตะกร้ามี Nolimit Level Up+ → ต้องแจ้งเตือนก่อนไป Step 3
→ ถ้าตะกร้ามีเฉพาะ G3D/Page → ข้ามขั้นตอนนี้ ไป Step 3 ได้เลย
→ ถ้าตะกร้ามี Nolimit + Page → ต้องแจ้ง (เพราะมี Nolimit)
```

ใหม่:
```
TRIGGER: หลัง Step 2 (Confirm) ลูกค้ายืนยันรายการแล้ว
→ ถ้าเป็น VIP (ตาม Memory Notes) → ข้ามขั้นตอนนี้เสมอ ไป Step 3 (Payment) ได้เลย ไม่ต้องแจ้ง
→ ถ้าในตะกร้ามี Nolimit Level Up+ → ต้องแจ้งเตือนก่อนไป Step 3
→ ถ้าตะกร้ามีเฉพาะ G3D/Page → ข้ามขั้นตอนนี้ ไป Step 3 ได้เลย
→ ถ้าตะกร้ามี Nolimit + Page → ต้องแจ้ง (เพราะมี Nolimit)
```

### จุดที่ 2: บรรทัด "รับ:" — ตัด branch VIP (VIP มาไม่ถึงขั้นนี้แล้ว)

เดิม:
```
รับ: ตกลง, โอเค, ได้, รับได้, ยอมรับ, ok
→ ลูกค้าทั่วไป: ไป Step 3 (Terms) ปกติ
→ VIP: ไป Step 3 (Payment) ข้ามข้อตกลง
```

ใหม่:
```
รับ: ตกลง, โอเค, ได้, รับได้, ยอมรับ, ok
→ ไป Step 3 (Terms) ปกติ
```

### จุดที่ 3: กฎ ⛔ ท้าย section

เดิม:
```
⛔ ทั้ง VIP และลูกค้าทั่วไป ต้องแจ้งเหมือนกัน
```

ใหม่:
```
⛔ VIP ข้ามขั้นตอนนี้เสมอ — ห้ามแจ้ง Support ล่าช้ากับ VIP
```

### สิ่งที่คงเดิม

- SCRIPT แจ้งเตือน, script ปฏิเสธ, script ถาม "นานเท่าไหร่" (ลูกค้าทั่วไปยังใช้; VIP ถามเรื่องระยะเวลาเอง บอทตอบตามจริงได้ปกติ)
- SCOPE เฉพาะ Nolimit Level Up+ (G3D/Page ไม่แจ้งอยู่แล้ว)
- `⛔ ห้ามข้ามขั้นตอนนี้: ถ้าตะกร้ามี Nolimit → ต้องได้ "ตกลง" ก่อนไป Terms/Payment` — คงไว้ แต่ใช้กับลูกค้าทั่วไปเท่านั้น (บรรทัด VIP ใหม่ใน TRIGGER ระบุข้อยกเว้นชัดแล้ว)

## ขั้นตอนปฏิบัติ (convention เดิมของโปรเจกต์)

1. Backup: INSERT prompt เดิมทั้งก้อนลง `flow_audit_logs` (แบบเดียวกับ audit id 8, 9, 14, 15, 17)
2. UPDATE `flows` SET `system_prompt` = ฉบับแก้ WHERE id=24 (แก้ 3 จุดข้างบน แบบ string replace ตรงเป้า)
3. Verify: SELECT ส่วน `<support_delay_warning>` กลับมาตรวจว่าครบ 3 จุด + ความยาว prompt เปลี่ยนสมเหตุสมผล + ไม่มีส่วนอื่นถูกกระทบ

## ความเสี่ยง

- ต่ำ: แก้ผิดจุด/กระทบส่วนอื่น → กันด้วย string replace เป๊ะ ๆ + verify + มี backup rollback ได้ทันที
- LLM ไม่ recognize VIP บางเคส → fallback เป็นพฤติกรรมเดิม (แจ้งเตือน) ไม่มีความเสียหาย
