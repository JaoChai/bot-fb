# Stock Count Injection — บอทรู้จำนวน stock จริง ไม่รับออเดอร์เกินของที่มี

**วันที่:** 2026-07-23
**สถานะ:** อนุมัติ design แล้ว รอ implementation plan

## ปัญหา

ระบบเช็ค stock ก่อนขายทุกชั้นเป็นแบบ boolean (มี/หมด) — ไม่มีชั้นไหนรู้ **จำนวน**:

- `StockInjectionService` ฉีดสถานะเข้า prompt แค่ "มีพร้อมส่ง / หมดชั่วคราว"
- `StockGuardService` (ด่านหลัง LLM) ก็เช็คแค่ `in_stock` true/false
- `stock:sync-pool` (ทุก 5 นาที) นับของจริงใน `items_available` อยู่แล้ว แต่ใช้แค่เงื่อนไข `> 0` เพื่อเปิด/ปิดสวิตช์ `in_stock`

ผลคือเคส "ลูกค้าสั่ง 10 แต่ stock มี 5" หลุดผ่านทุกชั้น: บอทรับออเดอร์ 10 → ลูกค้าจ่ายครบ → ตอนส่งจองได้แค่ 5 ชิ้น อีก 5 ชิ้นเป็น `shortage` เจ้าของต้องส่งเองในแชท และลูกค้าไม่รู้ตัวว่าได้ไม่ครบ

## เป้าหมาย

กันปัญหาตั้งแต่ **ต้นทาง** (ก่อนลูกค้าจ่ายเงิน): บอทรู้จำนวนคงเหลือจริง → ไม่รับออเดอร์เกิน stock → ลูกค้าสั่งเกินให้เสนอขายเท่าที่มี

**การตัดสินใจที่ล็อกแล้ว (จากเจ้าของ):**

1. ขอบเขต = ต้นทางอย่างเดียว (ไม่แตะระบบ delivery ปลายทาง)
2. บอทพูดเรื่องจำนวนคงเหลือกับลูกค้า **เฉพาะตอนลูกค้าสั่งเกิน** เท่านั้น — ขายปกติห้ามเผยตัวเลข
3. เมื่อสั่งเกิน → **เสนอขายเท่าที่มี** เช่น "ตอนนี้พร้อมส่ง 5 ชิ้นครับ รับ 5 ก่อนไหมครับ ส่วนที่เหลือของเข้าแล้วจะรีบแจ้งครับ"

## Design

### 1. เก็บตัวเลขคงเหลือ (DB + Sync)

- เพิ่ม column `available_count` (integer, nullable) ในตาราง `product_stocks`
  - สินค้า `delivery_method = 'stock'` ที่มี `stock_code` → เก็บจำนวนจริง
  - สินค้าอื่น (เช่น support_link) → null เสมอ ไม่เกี่ยวข้อง
- `SyncProductStockFromPool` (`stock:sync-pool` รันทุก 5 นาทีอยู่แล้ว) บันทึกตัวเลขจาก `countAvailable()` ลง `available_count` ทุกรอบ นอกเหนือจากการ toggle `in_stock` เดิม
- ถ้าตัวเลขของสินค้าใดเปลี่ยน → `Cache::forget(ProductStock::STOCK_CACHE_KEY)` เพื่อให้บอทเห็นเลขใหม่ในเทิร์นถัดไป
- **ไม่แตะ** `RagCache::purgeForProduct` — ยังล้างเฉพาะตอนสวิตช์ `in_stock` เปลี่ยนเหมือนเดิม (จำนวนไม่ได้ฝังอยู่ใน RAG cache)

### 2. ฉีดจำนวนเข้า prompt (StockInjectionService)

เพิ่มใน `buildStockInjection()` (ส่วนหัว prompt ที่ฉีดทุกเทิร์นอยู่แล้ว) สำหรับสินค้า in-stock ที่มี `available_count`:

```
[จำนวนพร้อมส่ง]: BM แดง = 5, เฟสส่วนตัว = 12
กติกา: ห้ามรับออเดอร์/เพิ่มตะกร้า/สรุปยอดเกินจำนวนพร้อมส่งเด็ดขาด
- ลูกค้าสั่งไม่เกินจำนวนพร้อมส่ง → ขายปกติ ห้ามพูดถึงจำนวนคงเหลือ
- ลูกค้าสั่งเกิน → เสนอขายเท่าที่มี บอกจำนวนที่พร้อมส่งตอนนี้ และแจ้งว่าส่วนที่เหลือ
  ของเข้าแล้วจะรีบแจ้ง (สรุปยอด/ราคาตามจำนวนที่ขายจริงเท่านั้น)
```

และเสริมบรรทัดสั้นๆ ใน `buildStockReminder()` (ท้าย prompt) ตาม pattern double-injection เดิมของโปรเจกต์ — LLM มักลืมกติกาที่อยู่ต้น prompt

- สินค้าที่ `available_count` เป็น null → ไม่ปรากฏใน `[จำนวนพร้อมส่ง]`
- สินค้าที่หมด (`in_stock = false`) → อยู่ในส่วน "หมดชั่วคราว" เหมือนเดิม ไม่โชว์เลข
- `injectStockStatus()` (emulator/test endpoints) ใช้ builder เดียวกัน ได้ผลอัตโนมัติ

### 3. สิ่งที่ไม่แตะ (ตั้งใจ)

- **ระบบ delivery ปลายทางทั้งหมด** — พฤติกรรม partial delivery + การ์ดเตือน Telegram + เตือนซ้ำ 30 นาที ยังเป็นตาข่ายสำรองเหมือนเดิม รองรับเคสตัวเลขหน่วง
- **StockGuardService** — ไม่เพิ่ม logic นับจำนวน (regex นับจำนวนซับซ้อนเกินคุ้ม) ถ้าเจอเคสบอทฝ่าฝืนกติกาจริงค่อยพิจารณาเพิ่มทีหลัง

### ข้อจำกัดที่ยอมรับ

- ตัวเลขหน่วงได้สูงสุด ~5 นาทีตามรอบ sync + cache TTL — เคสลูกค้าสองคนแย่งซื้อของชิ้นท้ายๆ พร้อมกันยังหลุดได้ แต่ตาข่าย delivery เดิมรับไว้ (จองไม่ได้ = shortage แจ้งเจ้าของ)
- พึ่ง LLM ทำตามกติกาใน prompt — มีโอกาสเผลอรับเกินได้บ้าง ยอมรับความเสี่ยงนี้เพราะมี double-injection ช่วยกดโอกาสให้ต่ำ และปลายทางมีตาข่ายรอง

## การทดสอบ

1. **Sync:** `stock:sync-pool` บันทึก `available_count` ถูกต้อง / สินค้า non-stock เป็น null / ล้าง `STOCK_CACHE_KEY` เมื่อเลขเปลี่ยน / ไม่ล้างเมื่อเลขเท่าเดิม
2. **Injection:** ข้อความมี `[จำนวนพร้อมส่ง]` + กติกาครบ / สินค้า null ไม่โชว์ / ของหมดไม่โชว์เลข / reminder มีบรรทัดจำนวน
3. **Manual หลัง deploy:** คุยผ่าน emulator สั่งเกินจำนวนจริง → บอทเสนอขายเท่าที่มีและไม่สรุปยอดเกิน
